# University Portfolio Notes

*Guidance for using this project in a university application (Common App
activities/essays, portfolio submissions, etc.). Write your own account of
your actual role and experience — do not copy these bullet points verbatim
into an application, as admissions readers value specificity and authentic
voice over generic phrasing.*

## What This Project Can Demonstrate

**Technical complexity**
- A 15-page React/TypeScript application with a shared design system,
  reusable component library, and 20+ distinct data visualizations
- Working with a modern build toolchain (Vite, Tailwind v4) and a
  charting library (Recharts) beyond tutorial-level use

**Initiative and scope**
- Independently defining the scope of a 14-module product rather than
  building a single tutorial project
- Writing full supporting documentation (architecture, business model,
  roadmap) — the kind of artifact real engineering teams produce, not just code

**Design thinking**
- Building an intentional visual identity (not a default template) suited to
  the subject matter — an "ops room" aesthetic for an environmental monitoring
  product
- Making UX decisions about how to present complex, multi-domain data
  (composite scores, gauges, ticker) to non-expert users

**Social/global awareness**
- Engaging seriously with climate, water, biodiversity, and disaster-risk
  topics — translating public science (temperature anomalies, AQI standards,
  reservoir capacity) into an accessible interface

**Entrepreneurial thinking**
- Producing a business model, pitch deck, and awards strategy alongside the
  code — showing the ability to think about a technical project as a product

## What to Be Careful About in an Application

- Be precise about what you personally built vs. what tooling (including AI
  assistance) helped generate. Admissions officers and scholarship committees
  value honesty about process.
- Don't claim real users, real deployed data pipelines, or measured impact
  that didn't happen — this repository runs on mock data, and your essay
  should say so if asked directly.
- Frame it as "I designed and built the foundation of a climate-tech platform"
  rather than "I built a global monitoring system," which the current
  repository does not do.

## Suggested Essay Angles

- The specific design decision you're proudest of (e.g., why a custom SVG map
  instead of a mapping API, why a composite gauge instead of a raw number)
- A technical problem you had to reason through (e.g., structuring one shared
  mock-data module so 14 pages stay consistent)
- What you'd build next if you kept working on it (tie to the Roadmap doc)
